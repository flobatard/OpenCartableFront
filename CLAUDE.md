# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Projet

Front Angular 22 d'**OpenCartable**, plateforme pédagogique auto-hébergée (AGPLv3) : un prof authentifié (OIDC/Zitadel) compose des cours, les élèves les consultent via liens publics sans compte. Cadrage produit : `Descriptions.md`. Design system (source de vérité UI) : `DESIGN_SYSTEM.md`.

## Commandes

Node est géré par nvm ; dans un shell non-interactif, npm n'est pas sur le PATH :

```bash
export PATH="$HOME/.nvm/versions/node/v26.3.0/bin:$PATH"
```

```bash
npm start                              # dev server (4200)
npm run build                          # build prod + prerender de la home
npm test                               # vitest (jsdom)
npm test -- --include src/app/core/theme/theme.service.spec.ts   # un seul fichier
npm test -- --filter "ThemeService"    # par nom de suite/test
npm run serve:ssr:OpenCartableFront    # sert dist/ via Express (4000)
docker compose up --build              # conteneur SSR (4000)
```

Après tout changement touchant SSR/i18n/prerender, vérifier que `dist/OpenCartableFront/browser/index.html` contient bien le texte français rendu (pas de chaînes vides) : c'est le symptôme d'une régression du préchargement Transloco.

## Architecture et invariants

**Zoneless** (pas de zone.js — défaut Angular v22) : tout état partagé est exposé en **signaux** ; les souscriptions rxjs écrivent dans des signaux. Ne jamais ajouter `provideZoneChangeDetection`.

**Trois modes de rendu** (`src/app/app.routes.server.ts`) : `/` prerendered au build, `/auth/callback` client-only (il manipule URL et storage), `**` SSR. Toute nouvelle route doit choisir son mode ; une route qui touche `window`/`localStorage` sans garde ne peut pas être SSR/prerender. Une route protégée par `authGuard` doit être en `RenderMode.Client` (le guard renvoie `false` au serveur — jamais authentifiée au SSR, donc jamais prerendable) : cf. `:lang/subjects`.

**Couche données HTTP** : `SubjectService` (`src/app/core/subjects/`) est le patron des accès API. Un seul `HttpClient.get` par ressource, mis en cache via `shareReplay({ bufferSize: 1, refCount: false })`, état exposé en signaux (`tree`/`loading`/`error`) alimentés par la souscription rxjs. Le Bearer Zitadel est attaché **automatiquement** par l'intercepteur d'`angular-oauth2-oidc` à toute requête sous `environment.apiUrl` (`resourceServer.allowedUrls` dans `app.config.ts`) — ne pas ajouter de header manuel ; `apiUrl` inclut déjà `/api`. Fetch déclenché côté navigateur uniquement (guard `isPlatformBrowser`). Les helpers purs de l'arbre (recherche, chemin d'ancêtres, aplatissement, lignes de treeview) vivent dans `subject.utils.ts`, testés isolément. `EducationLevelService` (`src/app/core/education-levels/`) est le second exemplaire du patron (mêmes signaux + cache) ; ses helpers minimaux vivent dans `education-level.utils.ts` — on ne généralise **pas** `subject.utils.ts` pour ne pas coupler les deux pickers. Les libellés métier (`nom` des matières et des niveaux d'étude) viennent de l'API et **ne passent pas** par l'i18n.

`UserProfileService` (`src/app/core/users/`) est la variante **mutable** du patron : pas de `shareReplay` figé mais un signal `profile` source de vérité, une **promesse en vol partagée** (`ensureLoaded()` — callback et guard n'émettent qu'un GET, invalidée sur erreur pour permettre le retry), les mutations remplacent le signal (`saveProfile()` : le `PUT /users/me/profile` sert d'onboarding initial **et** d'édition de profil, sémantique remplacement, pas de refetch) et un `effect` purge le profil quand `AuthService.isAuthenticated()` retombe. La logique du formulaire de profil (forme du FormGroup typé, règles de cohérence, mapping payload, complétude) est **factorisée en helpers purs** dans `core/users/profile-form.ts`, partagés par l'onboarding et la page profil — les templates, eux, restent distincts (stepper vs page à plat). Piège d'ordre : `patchFormFromProfile` pose `systeme` avant les blocs, sinon la cohérence câblée viderait les niveaux patchés.

**Espace cours** (`core/courses/` + `features/courses/`) : `CourseService` est la variante mutable **multi-ressources** du patron — signaux `list`/`detail` sources de vérité **refetchés à chaque entrée de page** (pas de `shareReplay` : données mutables), mutations async (`createCourse`, `addBlock` — qui porte un méta titre/description optionnel —, `deleteBlock`, `reorderBlocks`, `updateBlockContent`, `updateBlockMeta`) qui patchent localement le signal `detail` depuis la réponse du back (`#patchDetail` ne s'applique que si le détail affiché est le cours muté), purge à la déconnexion. Les blocs portent `titre`/`description` facultatifs (tous types, distincts du `content`). Helpers purs du formulaire de création dans `core/courses/course-form.ts` (motif `profile-form.ts`) ; ceux du méta titre/description d'un bloc dans `core/courses/block-meta-form.ts`, **partagés par la modale de création et l'éditeur**. Pages : liste `courses` (cartes, badges matières/niveaux résolus dans les arbres — id inconnu = pas de chip), création `courses/new` (**déclarée avant** `courses/:id`, matching en ordre ; picker de niveaux filtré par le `systeme_scolaire` du profil), espace blocs `courses/:id` (liste = titre en gras + description ; **création via une modale** `block-create-dialog` saisissant titre/description, puis **redirection** vers l'éditeur du bloc créé ; « Modifier » sur **tous** les types ; supprimer/réordonner par boutons ; type `ressource` visible mais désactivé tant que l'upload S3 n'existe pas ; suppression en deux temps sans modale, désarmée au blur), éditeur de bloc `courses/:id/blocks/:blockId`. Toutes protégées `[authGuard, onboardingGuard]` + `RenderMode.Client`, params lus en snapshot (pas de `withComponentInputBinding` dans ce projet). Le header porte le lien « Mes cours » quand la session est ouverte.

**Éditeur Monaco** (`shared/markdown-editor/`, CVA `string` enrobant `ngx-monaco-editor-v2`) — pièges vérifiés dans le source du wrapper, ne pas « simplifier » :
- Monaco est **servi en AMD depuis les assets** (`angular.json` copie `node_modules/monaco-editor/min/vs` → `/monaco/vs`, ~16 Mo), jamais bundlé ; `NGX_MONACO_EDITOR_CONFIG` est fourni par le composant avec un **baseUrl absolu `/monaco/vs`** (le défaut du wrapper est relatif à la page → cassé sur les routes profondes). Workers et CSS monaco s'auto-configurent depuis ce chemin.
- Le wrapper **dispose et recrée l'éditeur à chaque changement de référence de `[options]`** → options figées en constante module ; le thème passe par `monaco.editor.setTheme` global (effect sur `ThemeService.theme()`), **jamais** par les options.
- Le wrapper **n'est pas SSR-safe** (touche `window`/`document` sans guard) → toute page hôte en `RenderMode.Client` + `@if (isBrowser)` dans le composant.
- Relais CVA avec **guard anti-écho** (le wrapper ré-émet la même valeur après un `writeValue`) ; `remeasureFonts()` quand JetBrains Mono finit de charger (sinon curseur décalé).
- En jsdom (specs), le loader AMD ne se charge jamais → le wrapper est **inerte par construction** : les specs pilotent les `FormControl`, jamais monaco.

**Champ markdown réutilisable** (`shared/markdown-field/`, CVA `string`) : composite qui embarque les **onglets Éditeur | Aperçu** (tablist APG), l'`app-markdown-editor` (Monaco), l'**aperçu** (rendu local `renderCourseMarkdown` + Mermaid dans un `effect` gardé sur l'onglet aperçu **et** `isPlatformBrowser`) et la modale d'aide. C'est l'unité que composent les éditeurs de contenu — bloc texte (une instance) et éditeur d'exercice (plusieurs : sujet + un énoncé par question ; la réponse attendue reste un textarea simple, décision produit). Composition d'un `FormControl` interne (patron `SubjectMultiPicker`) exposé **public** `control` (exception à la convention `protected` : jsdom ne peut pas taper dans monaco, les specs pilotent le contrôle) ; panneau éditeur masqué par **`[hidden]`, jamais `@if`** (un `@if` rechargerait monaco à chaque bascule) ; ids ARIA du tablist **uniques par instance** via compteur de module (**jamais** `Date.now()`/`Math.random()`). Usage **Client-only** (rendu DOMPurify + Monaco touchent `window`). `MarkdownHelpDialog` (`shared/markdown-help-dialog/`) est la modale d'aide Markdown/LaTeX/Mermaid extraite : `<dialog>` natif présentational, méthodes publiques `open()`/`close()`, ref `#dialogEl` (**pas** `#dialog` : une ref de template ne doit pas collisionner avec le signal `viewChild` `dialog`). Onglets et pilules d'action sont des **utilitaires globaux** (`.tabs`/`.tab`/`.tab--active`, `.pill`/`.pill--active`) dans `_components.scss`, à côté de `.btn` (cf. `DESIGN_SYSTEM.md §8`). Les clés i18n de ces composants partagés vivent sous `markdownField.*` / `markdownHelp.*` (et **non** `courses.editor.*`).

**Éditeur de bloc** (`features/courses/block-editor/`) : **coquille-page** (en-tête, **formulaire titre/description à enregistrement explicite — tous types**, indicateur d'autosave du contenu, espace de travail redimensionnable éditeur/assistant par poignée `separator` ARIA + pointer capture, états de chargement). Le **méta** (titre/description) est un `buildBlockMetaForm` séparé, initialisé une fois depuis le bloc, bouton « Enregistrer » actif seulement si **modifié** (snapshot JSON du payload, motif page profil) ; `saveMeta` appelle `updateBlockMeta` qui n'envoie **que** le méta (jamais le `content`), pas de flush au `ngOnDestroy` (save explicite, non autosave). Le **contenu** est délégué par type : `texte` → composite `MarkdownField` (`<app-markdown-field [formControl]="content">`), `exercice` → `ExerciseEditor` (`features/courses/exercise-editor/`, présentationnel : **deux onglets Sujet | Questions** — tablist APG au motif `markdown-field`, ids ARIA par compteur de module, panneaux masqués par `[hidden]` jamais `@if` (Monaco vit dans les deux) — ; `[initial]` lu une seule fois, `(contentChange)` remonte chaque frappe ; helpers purs `core/courses/exercise-form.ts`). **Autosave** conservé ici, dans un **pipeline unique** : `merge(content.valueChanges, frappes exercice) → debounceTime(1500) → concatMap` (**jamais `switchMap`** : les promesses de PATCH ne sont pas annulables, une réponse périmée écraserait la plus récente), signal `saveState` (`idle|dirty|saving|saved|error` — le flux survit aux erreurs, retentative à la frappe suivante), init **une seule fois** depuis le bloc chargé et jamais réécrit depuis le signal (le patch du détail post-save ne doit pas écraser la frappe), flush fire-and-forget au `ngOnDestroy`. **Deux invariants exercice** : le payload est construit **à l'envoi du PATCH** (état courant du formulaire), jamais à l'émission, et les ids de questions générés par le back sont **réécrits après chaque save** (`applyGeneratedIds` sur un snapshot des groupes capturé à l'envoi) — sinon l'autosave suivant renverrait `id: null` et le back régénérerait des ids censés être stables à vie (contrat J2). Le contrôle `content` et le `form` de l'ExerciseEditor sont publics (exception à la convention `protected`) : jsdom ne peut pas taper dans monaco, les specs les pilotent.

**Panneau assistant & espace de travail redimensionnable** (`features/courses/course-chat/` + layout du block-editor) : `CourseChat` (`app-course-chat`) est la **coquille présentationnelle placeholder** de l'assistant IA — en-tête + pastille « bientôt », zone de messages `role="log"` en état vide, saisie **désactivée** ; tokens du DS, clés i18n **`courseChat.*`**, **aucun** HTTP/oauth. Le câblage IA futur (service + signal `messages`/`send()`) est **additif** : les inputs `courseId`/`blockId` sont déjà passés (réservés) et l'output `collapse` remonte le repli au parent. Le block-editor la monte dans un **espace de travail flex à deux colonnes** (`app-markdown-field` | poignée | `app-course-chat`) — **premier layout à colonnes et premier breakpoint (`@media` 900px) du projet**. Invariants à préserver : (1) la largeur de la colonne éditeur vient du signal `editorPct` (borné 15–85 %) via la **custom property `--editor-basis`**, **jamais** un `flex-basis` inline (sinon les états CSS ne peuvent plus la surcharger) ; (2) le **repli** (`chatCollapsed`) masque chat + poignée par **`[hidden]`, jamais `@if`** (préserver la future conversation) et `.block-editor__workspace--solo` fait **grandir** l'éditeur (`flex:1`) à leur place ; (3) la poignée redimensionne via `setPointerCapture` (monaco ne vole pas les events pendant le glissé), axe dérivé du `flex-direction` réel (row → X, column → Y), clavier flèches/Home/End. Le plafond du shell `.app-main` est **relevé (1800px) pour cette seule page** (les autres pages gardent leur propre max-width ≤ 880). Sous **900px** : empilement vertical, **drag conservé** (axe Y, hauteur définie maintenue — l'IA peut être verbeuse) et un **bouton de réouverture** réapparaît sous l'éditeur (`display:none` sur desktop, où la bascule « Assistant » de la barre d'outils suffit) pour rouvrir le chat sans remonter. Libellés des contrôles sous `courses.editor.{resize,toggleChat,showChat}`.

**Rendu markdown de cours** (`core/markdown/course-markdown.ts`) : `renderCourseMarkdown` = instance `Marked` dédiée (ne **jamais** muter le singleton `marked`, son `use()` est global) + extension math maison (`$…$` en ligne, `$$…$$` centré — KaTeX `throwOnError: false`, LaTeX invalide rendu en `.katex-error`) + **DOMPurify** (profils html+mathMl+svg, `ADD_TAGS: semantics/annotation` — jamais `annotation-xml`, vecteur mXSS). **C'est l'unique point de sanitisation et de bypass du HTML de cours** : les consommateurs (aperçu du champ markdown, future vue élève) injectent sa sortie via `bypassSecurityTrustHtml` — justifié car le sanitizer d'Angular dépouillerait les `style` inline et le MathML/SVG de KaTeX — et ne bypassent jamais autre chose. ⚠ Sans `window` (SSR), DOMPurify retourne le HTML **non filtré** : consommation Client-only obligatoire. `katex.min.css` est chargé globalement (angular.json, fontes copiées au build) ; `katex`/`dompurify` sont des dépendances directes, alignées sur les versions transitives de mermaid.

**Composants partagés** (`src/app/shared/`) : composants réutilisables transverses. `SubjectPicker` y est le premier `ControlValueAccessor` (valeur = `id` du nœud, utilisable en Reactive Forms) ; tout nouveau form control suit ce modèle (provider `NG_VALUE_ACCESSOR` + `forwardRef`, `writeValue`/`registerOnChange`/`setDisabledState`, navigation clavier + ARIA treeview). `EducationLevelPicker` est le premier CVA **multi**-sélection : valeur = `string[]` d'ids toujours émise en ordre d'arbre (jamais `null`, ids inconnus préservés) ; pattern volontairement plus simple — disclosure + cases à cocher natives (pas de treeview ARIA ni de recherche, arbre ~22 nœuds toujours déplié), chips « badge niveau » (slate) sous le champ. `EducationLevelPicker` accepte un input `systeme` (filtre les **racines** de l'arbre au système scolaire donné ; `null` = tous) — le contrat « ids inconnus préservés » demeure : c'est au parent de vider la valeur quand le système change. `SubjectMultiPicker` (`shared/subject-multi-picker/`) est le CVA multi-matières **par composition** : il embarque le `SubjectPicker` mono (non modifié) comme champ d'ajout via un `FormControl` interne réinitialisé à chaque sélection, valeur = `string[]` en ordre d'ajout dédoublonné, chips « badge matière » (indigo). **Piège de nommage** : `SubjectLevel` et les clés `subjects.level.*` désignent la *profondeur* de l'arbre des matières ; tout ce qui touche aux niveaux d'étude vit sous `EducationLevel*` / `educationLevels.*`. Pièges : un `viewChild()` ne peut pas être posé sur un champ privé ES (`#`) — le marquer `protected` ; et ne pas nommer une variable de template (`#foo`) comme un signal du composant (le template résout la ref avant le signal).

**Notifications globales & spinner** (`core/notifications/` + `shared/snackbar/` + `shared/spinner/`) : `NotificationService` est la variante **in-memory** du patron à signaux (pas de HTTP) — file `toasts` source de vérité, API `error/warning/info/success/dismiss`, ids par compteur incrémental (**jamais** `Date.now()`/`Math.random()`), auto-fermeture par `setTimeout` gardée `isPlatformBrowser` (aucun timer au SSR), dédoublonnage des messages identiques déjà visibles ; les émetteurs poussent un `message` **déjà traduit** (le service et le composant ignorent l'i18n, comme les libellés métier). `Snackbar` (monté **une seule fois** dans `app.html`, ancré bas-gauche, `z-index` élevé, `pointer-events: none` sur le conteneur / `auto` sur les toasts) rend la file : `role="alert"` pour une erreur / `role="status"` sinon, liseré gauche de la teinte **500** sémantique + glyphe distinctif (l'info n'est **jamais** encodée par la seule couleur, et 500 reste un aplat, jamais du texte). Câblage actuel : **seul `AuthService`** l'alimente (toast `loginError` quand la discovery échoue au login, `connectionError` quand le refresh silencieux échoue au bootstrap) — `AuthService` reste seul importeur d'`angular-oauth2-oidc`, il ne fait qu'appeler ce service applicatif ; brancher les autres sources (HTTP, données) plus tard sans refonte. `Spinner` (design system, `role="status"` + label `.sr-only` « Chargement… » par défaut, tailles `sm/md/lg`, anneau CSS coloré par `--color-primary`, rotation figée en `prefers-reduced-motion` → repli statique) recouvre l'éditeur Monaco tant que `ready()` (ex-`#ready`, exposé en lecture) est faux dans le markdown-editor. Les clés vivent sous `notifications.*` et `common.loading` (fr/en symétriques) ; l'utilitaire `.sr-only` est global (`_base.scss`). Doc UI : §8 de `DESIGN_SYSTEM.md`.

**Chaîne i18n (fragile, ne pas "simplifier")** — Transloco fr/en :
- Le loader (`src/app/core/i18n/transloco-loader.ts`) passe par des `import()` dynamiques de JSON, **pas** par HTTP : au prerender il n'y a aucun serveur pour répondre à une URL relative.
- Un `provideAppInitializer` dans `app.config.ts` précharge la langue active avant le premier rendu — sans lui, le HTML prerendered sort avec des chaînes vides.
- La langue persistée de l'utilisateur est restaurée **après** l'hydratation (`afterNextRender` dans `App`) : la restaurer plus tôt fait diverger le premier rendu client du DOM serveur (erreur NG0500).
- Traductions dans `src/app/i18n/{fr,en}.json` — garder les deux fichiers symétriques.

**Thème clair/sombre** : attribut `data-theme="dark"` sur `<html>`, posé avant le premier paint par le script inline d'`index.html` (anti-FOUC). `ThemeService` **lit** cet attribut au bootstrap au lieu de recalculer — préserver ce contrat. Les swaps visuels dépendant du thème (ex. logo) se font en CSS sur `[data-theme]`, jamais par binding (mismatch d'hydratation).

**OIDC** : `AuthService` (`src/app/core/auth/`) est la **seule** couche autorisée à importer `angular-oauth2-oidc` (exigence de remplaçabilité de l'IdP, Descriptions.md §8). Code flow + PKCE, client public. Init browser-only et sans appel réseau au démarrage : les pages publiques élèves ne doivent jamais dépendre de la disponibilité de Zitadel. Le `state` restauré au callback est validé (chemins internes uniquement).

**Onboarding bloquant** (`features/onboarding/`, route `:lang/onboarding`, RenderMode.Client) : au retour du callback (`AuthCallback`), le profil est chargé (`ensureLoaded()`) et un profil incomplet est redirigé vers `/:lang/onboarding?next=<cible>`. Les routes protégées portent `canActivate: [authGuard, onboardingGuard]` — **dans cet ordre** (le guard onboarding laisse passer les non-authentifiés), et `onboardingGuard` ne doit **jamais** être posé sur la route onboarding elle-même (boucle) : c'est le composant qui renvoie un profil déjà complet vers `next`. Guard et callback sont **fail-open** si l'API profil est injoignable (ne jamais enfermer l'utilisateur hors de l'app). La page est le premier usage des **Reactive Forms typés** du projet : en zoneless, la réactivité du template passe par `toSignal(form.valueChanges)` ; le stepper est **dynamique** (liste d'étapes `computed` dérivée des rôles cochés — prof, élève ou les deux ; décocher un rôle vide son bloc, changer de système vide les niveaux choisis).

**Page profil** (`features/profile/`, route `:lang/profile`, `canActivate: [authGuard, onboardingGuard]`, RenderMode.Client) : mêmes sections que l'onboarding à plat, pré-remplies via `ensureLoaded()` + `patchFormFromProfile`, sauvegarde par `saveProfile()`. « Enregistrer » n'est actif que si le formulaire est **complet ET modifié** — détection par snapshot JSON du payload (déterministe : niveaux en ordre d'arbre, matières en ordre d'ajout). **Menu utilisateur** (`layout/header/user-menu.*`) : pattern APG menu-button (déclencheur = nom de l'utilisateur, `aria-haspopup="menu"`, panneau `role="menu"`, Escape → refocus déclencheur, focusout externe → ferme, flèches ouvrent/cyclent) ; « Se déconnecter » vit dans ce menu, plus dans le header. Zoneless : le focus post-ouverture passe par `afterNextRender(..., { injector })` car le panneau n'est pas encore rendu au `set(true)`.

**Environnements** : `environment.ts` (défaut dev) remplacé par fileReplacements (`.development.ts` / `.prod.ts`). Le type commun vit dans `environment.model.ts`, qui n'est **jamais** remplacé — ne pas importer de types depuis `./environment` dans les fichiers remplacés (import circulaire au build). Config figée au build (pas d'injection runtime), sauf `ALLOWED_HOSTS` / `TRUST_PROXY_HEADERS` / `PORT`, lues par `src/server.ts` (protection SSRF d'Angular SSR : sans `ALLOWED_HOSTS` correct, le serveur répond 400).

**Design system** : les tokens CSS de `src/styles/_tokens.scss` recopient le §10 de `DESIGN_SYSTEM.md` — toute couleur/typo passe par `var(--…)`, jamais de hex en dur dans les composants. Cibles de contraste : AA partout, AAA dans `.course-content`. Pièges connus : `slate-400` interdit pour du texte (réservé désactivé/décoratif), l'ambre n'est jamais une couleur de texte, texte blanc sur indigo uniquement sur indigo-600/700. Polices auto-hébergées (@fontsource, déclarées dans `angular.json > styles` — pas d'`@import` Google Fonts). Libellés en sentence case (§9).

**Sass** : `styles.scss` n'utilise que `@use` (jamais `@import` scss, déprécié — et `@use` doit précéder toute autre règle).

## Tests

Vitest via le builder `@angular/build:unit-test`, specs colocalisées `src/**/*.spec.ts`. `src/test-setup.ts` stubbe `matchMedia` et les storages (absents de l'environnement jsdom du builder). Pour les composants utilisant Transloco, passer par `provideTranslocoTesting()` (`src/app/testing/transloco-testing.ts`) ; mocker `AuthService` avec des signaux, `OAuthService` avec des `vi.fn()`. Attention : jsdom expose `navigator.language = 'en-US'`, ce qui influence les tests de `LanguageService`.

Services HTTP : `provideHttpClient()` + `provideHttpClientTesting()`, assertions via `HttpTestingController` (sans réseau réel). Les composants consommant un service de données le mockent avec des signaux (`tree`/`loading`/`error`) et des `vi.fn()` — pas de HTTP. Fixture d'arbre partagée : `src/app/testing/subjects.fixture.ts` (profondeur mixte, pour les helpers et le rendu). `--include` attend des chemins de specs (`*.spec.ts`) — un glob `dossier/**` embarque aussi `.html`/`.scss` et casse le build de test.
